# MD Explorer — Project Technical Profile for Resume

## Project Overview
**MD Explorer** is a fast, offline-first Markdown file explorer and viewer built with Flutter, designed for desktop and mobile platforms. It enables users to browse local markdown files, organize them into collections, import files into a personal library, and render them with full GitHub Flavored Markdown (GFM) support — all without requiring network connectivity.

## Architecture Patterns

### Layered MVVM Architecture
```
Models (Data Layer) → Services (Business Logic + Persistence) → Riverpod Providers (State Bridge) → Screens/Widgets (Presentation Layer)
```

- **Model-View-ViewModel (MVVM)** separation with Riverpod as the reactive state bridge
- **Service Layer Pattern**: Stateless, long-lived service classes encapsulating all I/O, persistence, and file-watching logic
- **Repository-like Providers**: Riverpod providers wrap services and expose state as `AsyncValue<T>` for unified loading/error/data handling
- **Consumer Widget Pattern**: All UI components extend `ConsumerWidget` or `ConsumerStatefulWidget` for reactive Riverpod consumption

### Reactive State Management (Riverpod)
- **StateNotifierProvider**: Manages all mutable state (settings, bookmarks, file tree, UI state, lists, imported files)
- **StateProvider**: Lightweight reactive state (selected file, current folder files)
- **FutureProvider.autoDispose**: Auto-disposing async file content loading with automatic memory release on selection change
- **Provider**: Read-only singleton service injection
- **AsyncValue Pattern**: Consistent `when(data/loading/error)` pattern across all UI layers

### Responsive Design Architecture
- **Three-tier breakpoint system**: Mobile (<600px), Tablet (600–1024px), Desktop (>1024px)
- **Platform-aware layout**: Desktop uses 3-panel row layout (sidebar + file list + viewer), Tablet uses 2-panel with toggle, Mobile uses single panel with bottom navigation + drawer
- **Custom breakpoint utilities**: `AppBreakpoints`, `LayoutMode` enum, `AppPlatform` detection (isDesktop, isMobile, hasMouse)

## Key Engineering Decisions & Best Practices

### Performance Optimizations
1. **Shallow Lazy Loading**: No recursive directory scanning at startup. Subdirectories loaded on-demand when user expands them — instant startup regardless of folder size
2. **LRU Cache**: In-memory directory listing cache (500 entries, 60s TTL) with path-prefix-based invalidation on file system changes
3. **Quick Markdown Detection**: `_folderHasMarkdownShallow()` checks only immediate + one sublevel for .md files, avoiding expensive full recursion
4. **AutoDispose Pattern**: `FutureProvider.autoDispose` releases file content memory when selection changes
5. **Parallel Root Loading**: `Future.wait()` loads all root folders concurrently
6. **Deferred Startup**: `addPostFrameCallback` defers folder loading until after first frame, preventing jank
7. **Error-per-Entity Isolation**: Individual inaccessible files/directories are skipped rather than failing entire operations

### Error Handling
- **Graceful Degradation**: All service methods return empty collections/null on failure rather than throwing
- **AsyncValue Propagation**: Errors flow through `AsyncValue.error(e, st)` to UI for consistent display
- **Protected Directory Detection**: System directories (Windows/, Program Files/, Android/, /data/, /system/) detected via regex and skipped preemptively
- **Batch Import Resilience**: Individual file import failures are skipped without aborting the entire batch

### Cross-Platform Compatibility
- **Android Scoped Storage**: Imported files copied to internal app storage (`md_library/`) to survive SAF permission revocation
- **Android 10–13+ Compatibility**: `requestLegacyExternalStorage` for Android 10, `enableOnBackInvokedCallback` for Android 13+ back gesture
- **Desktop Window Management**: Conditional `window_manager` initialization (Windows/macOS/Linux only), skipped on mobile/web via `kIsWeb` + `defaultTargetPlatform` checks
- **Runtime Permissions**: `permission_handler` for Android `READ_EXTERNAL_STORAGE` and `MANAGE_EXTERNAL_STORAGE`

### Code Quality
- **flutter_lints** with strict default rules, zero custom overrides
- **Riverpod code generation** tooling available (`riverpod_generator` + `build_runner`)
- **Type-safe models** with explicit serialization/deserialization
- **Immutable state updates** via `copyWith` pattern throughout

## Business Functionality

1. **Local Markdown Browsing**: Navigate local file systems, browse folders, preview markdown files
2. **Markdown Library**: Import external .md files into a personal, app-managed library with persistence
3. **Folder Collections**: Organize folders into named lists ("playlists") for project-based organization
4. **Bookmarking**: Star frequently-accessed folders/files for instant navigation
5. **Offline-First**: Zero network dependencies — all parsing, rendering, and persistence is local
6. **Cross-Platform**: Single codebase targeting Windows, macOS, Linux, Android, iOS

## Notes for Resume Customization

1. **Prioritize by role**: For Flutter-specific roles, emphasize Riverpod, responsive design, and cross-platform work. For generalist roles, emphasize architecture, performance, and problem-solving.
2. **Quantify where possible**: Add metrics if available (e.g., "handled 10,000+ file trees", "startup time < 200ms", "supporting 5 platforms from single codebase").
3. **Tailor bullet count**: Use 4-6 bullets for a single project on a resume; save the rest for interviews.
4. **Action verbs**: All bullets start with strong action verbs (Architected, Designed, Implemented, Engineered, Developed, Built, Applied, Configured, Established).
5. **STAR format implied**: Each bullet contains Situation/Task (the problem), Action (what you did), and Result (the benefit) in compressed form.
